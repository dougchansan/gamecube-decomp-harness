Knowledge Resource Graph Design

A proposed resource registry, vertical-slice indexer model, file graph, worker lookup API, and graph-ranked scheduling layer for the Melee decomp orchestrator.

status: design draft scope: decomp-orchestrator/knowledge primary users: director + worker agents api posture: CLI-first north star: matched_code_percent

1. Purpose

The orchestrator should stop treating knowledge as a loose list of good files to search. It should expose a structured resource system where each corpus has a stable API, indexer, freshness policy, trust level, and graph contribution contract.

The core product of this system is a file graph: for every source file, the orchestrator can answer what is known, where it came from, what is stale, what is newly useful, and whether the file is eligible for worker edits. Past PRs are the backbone of this graph, and other resources attach evidence and search surfaces around it.

Design target. Workers should call resource APIs like search_discord_knowledge, search_pr_history, or file_graph_card instead of manually wandering across raw folders. The director should consume graph-derived scoring features instead of only local objdiff closeness.

3. Expected Lifecycle

The knowledge graph grows in passes. It starts with a complete pull of the PR library, then performs a synthesis pass that links PRs, files, diffs, comments, reviews, and postmortems. After that, workers and merged PRs keep adding evidence through a curator layer before the SQLite graph is rebuilt.

1. Pull PRs Fetch the raw PR dump and vertical slices: metadata, changed files, comments, reviews, diffs, and postmortems.
2. Synthesize Run pending-only PR postmortem indexing; live background maintenance queues bounded pending batches to the PR-review agent by default. Then group changes by file, symbol, subsystem, review issue, tactic, and repeated mismatch pattern.
3. Build Cards Produce file graph cards that summarize known PR history, resource links, risks, fresh facts, and editability.
4. Work Targets Workers begin from the file card, then query PRs, Discord, data sheets, PDFs, or tool caches only when a concrete question appears.
5. Fold Back Worker reports, accepted changes, merged PRs, and refreshed resources flow through kg-maintain: PR index, curate, optional curator-agent proposal review, and graph rebuild.

Accumulating Knowledge Loop

  1. Raw resources are indexed into versioned records.
  2. The graph synthesis pass turns records into file, function, PR, document, and tool entities.
  3. The director asks for ranking features from the graph before filling the queue.
  4. A worker claims a target and automatically starts from the file graph card.
  5. The worker asks resource APIs for focused evidence instead of browsing raw folders.
  6. Worker output records attempted hypotheses, accepted facts, rejected facts, local validation, and retained edits in SQLite.
  7. The knowledge curator reduces worker reports and PR postmortems into graph-owned accepted lessons or proposal-only source updates.
  8. When upstream PRs merge, the PR refresh/index/synthesis pass links that merged work back into the same graph.
  9. Files connected to newly accepted facts or merged PRs receive fresh-edge signals for the next ranking cycle.
Operating model. The graph does not need to be perfect on day one. It should improve monotonically: every PR pull, worker report, accepted fact, and merged upstream change adds linkages that make future workers less blind.

4. Core Terms

Resource A durable input source such as past PRs, Discord knowledge, a PDF page index, the data sheet CSVs, or a Ghidra cache.
Vertical Slice The full package for one resource: descriptor, status API, refresh API, indexer, search API, result schema, and graph contribution rules.
Entity A node in the graph: source file, function, symbol, address range, PR, review comment, document chunk, struct, field, or tool pattern.
Fact A typed claim about an entity with evidence, confidence, trust tier, and resource version.
Edge A relationship between entities, such as TOUCHED_BY_PR, MENTIONS, CALLS, ANALOGOUS_TO, UNLOCKS, or BLOCKED_BY.
File Graph Card The query result that summarizes everything known about one source file and whether it can be edited by workers.

6. Standard Resource API

The API should be small enough for agents to understand and strict enough that every result can be traced back to evidence.

V1 API decision. Keep this as CLI-style commands with simple arguments. Do not wrap these resources in a full MCP server for v1. If Pi-agent extension/function access later wants to expose these calls directly, it should wrap the same CLI or TypeScript library entry points instead of creating a second API model.

  | "pr_corpus"
  | "document"
  | "csv_corpus"
  | "external_mirror"
  | "tool_output";

  | "canonical"
  | "local"
  | "reference"
  | "historical"
  | "external_hint"
  | "tool_evidence";

  output_tables: string[];
};

type SearchRequest = {
  query?: string;
  target?: {
    source_path?: string;

    | "naming"
    | "source_shape"
    | "compiler_behavior"
    | "address_lookup"
    | "review_risk"
    | "scheduling";

  resource_id: string;
  result_id: string;
  title: string;
  snippet: string;

CLI Surface

Workers and operators should be able to call the same surface from the shell while the TypeScript runtime uses the underlying library directly. Arguments should stay boring: resource ID, source path, symbol, query, limit, and JSON output.

7. Vertical Slice Contract

A resource should have the smallest useful vertical slice: descriptor, index command, search command, and graph-emission rules when the resource is important enough to link. The goal is that Discord, PDFs, PRs, and tool caches are callable in the same way without forcing heavyweight infrastructure around low-confidence external sources.

Descriptor Declares paths, trust, capabilities, freshness, refresh commands, and index outputs.
Indexer Builds deterministic JSONL/SQLite records from the resource input and writes a resource version hash.
Search API Accepts a standard request and returns bounded, cited results with resource IDs and entity IDs.
Graph Emitter Converts resource records into entities, facts, and edges when the resource is useful enough to link. Every edge has provenance.
Freshness Policy Describes whether the resource is static, generated from current repo state, refreshable from upstream, or live/on-demand.
Agent Affordance Describes when a worker should use the resource, how much to trust it, and what verifier should follow it. 
type ResourceSlice = {
  descriptor: ResourceDescriptor;
  status(): ResourceStatus;
  refresh?(): ResourceRefreshResult;
  index(): ResourceIndexResult;
  search(request: SearchRequest): SearchResult[];
  emitGraph(): {
    entities: GraphEntity[];
    facts: GraphFact[];
    edges: GraphEdge[];
  };
};

10. PRs As The Backbone

Past PRs carry the richest file-level history: exact changed paths, diff lines, review feedback, contributor rationale, bot reports, and generated postmortems. The code graph should ingest this first.

Secondary Synthesis Pass

Fetching PRs is only the acquisition step. A second pass should bring the corpus together by asking what changed, which files were touched together, which review comments corrected the same kind of issue, and which source tactics repeatedly produced accepted fixes. 

type PrSynthesisRecord = {
  pr: number;
  merged_at?: string;
  touched_files: string[];
  touched_symbols: string[];
  inferred_subsystems: string[];
  review_risks: string[];
  successful_tactics: string[];
  failed_tactics: string[];
  file_edges: Array<{
    source_path: string;
    edge_type:
      | "TOUCHED_BY_PR"
      | "REVIEWED_IN_PR"
      | "FIX_PATTERN_SEEN_IN_PR"
      | "FAILED_PATTERN_SEEN_IN_PR";
    evidence_ref: string;
  }>;
};
PR Artifact Graph Contribution
changed_files.jsonl Create TOUCHED_BY_PR edges from source files to PRs and derive subsystem heat.
diff_lines.jsonl Create source-shape, naming, macro, and field-use hints tied to exact files and hunks.
review_comments.md/json Create review-risk facts such as rejected names, style corrections, and unsafe patterns.
text_corpus.jsonl Create searchable rationale and tactic chunks associated with PRs, files, and symbols.
postmortem.json Create distilled lessons, successful tactics, failed tactics, and reusable scheduling hints.
Implementation priority. Build the PR-to-file graph before adding embeddings or complex document RAG. It is deterministic, already local, and directly feeds worker context and director ranking.

11. Worker Interaction

A worker target packet should include API affordances, not a giant dump of all available resources. The worker uses APIs to ask for more context when the next question is grounded.

Target Packet Additions

{
  "knowledge_api": {
    "file_card": {
      "tool": "file_graph_card",
      "args": {
        "source_path": "src/melee/it/items/itlinkbomb.c"
      }
    },
    "allowed_resources": [
      "code_graph",
      "past_prs",
      "ssbm_data_sheet",
      "discord_knowledge",
      "powerpc_docs",
      "external_tools"
    ],
    "allowed_tools": [
      "search_resource",
      "search_pr_history",
      "search_discord_knowledge",
      "tool_lookup"
    ],
    "budgets": {
      "resource_searches": 12,
      "tool_lookups": 4,
      "large_document_chunks": 8
    }
  }
}

Worker Loop

  1. Read the target packet and automatically call file_graph_card for the leased file.
  2. Review graph-selected adjacent files, recently updated linked files, and high-value PR/resource edges.
  3. Inspect local source, headers, objdiff metadata, and current match status.
  4. Use resource APIs for specific questions: naming, source shape, compiler behavior, address or offset lookup, similar PRs, or review risk.
  5. Turn resource hits into one or two source hypotheses with explicit evidence.
  6. Edit only paths in the lease write set.
  7. Verify with narrow builds and objdiff checks.
  8. Report progress, reusable facts, negative results, and any new graph edges.
Trust rule. Resource hits are not source truth. Local source, headers, symbols, splits, assembly, and objdiff verification outrank Discord, PR summaries, data sheets, and external mirrors.

Automatic File-Graph Boot

The worker should not decide from scratch where to search. The target packet should make the first lookup automatic and bounded: get the file card, inspect graph-ranked linked files and PRs, then decide which resource API answers the next specific question.

12. Orchestrator Interaction

The orchestrator uses the same knowledge layer differently. It does not need long snippets; it needs fresh, scored features that improve target selection.

Ranking Inputs

candidate_prior =
  objdiff_finishability
  + graph_degree_bonus
  + fresh_fact_bonus
  + pr_similarity_bonus
  + linked_unlock_bonus
  + duplicate_shape_bonus
  + worker_context_quality_bonus
  - recent_stall_penalty
  - stale_resource_penalty
  - reviewability_risk_penalty
  - data_layout_risk_penalty

Graph Ranking API

type GraphRankFeature = {
  target: {
    source_path: string;
    unit: string;
    symbol: string;
  };
  editability: "editable" | "read_only_complete" | "locked" | "blocked";
  graph_degree: number;
  fresh_edges_since_last_attempt: number;
  relevant_pr_count: number;
  review_risk_count: number;
  duplicate_reference_count: number;
  linked_unlock_potential: number;
  stale_fact_count: number;
  explanation: string[];
};

The board snapshot can merge these features with the existing objdiff candidate window. The director then receives candidates that are not only close to matching, but also newly constrained by facts, PRs, or similar files.

V1 ranking decision. Start deterministic. Use hand-tuned weights and record each feature value so later runs can show which signals actually predicted progress. A learned weighting layer can come after the graph has enough outcomes to learn from.

13. Editability And 100 Percent Files

Exact matches are evidence. They should normally be read-only for workers. The system should enforce this in the graph layer, target generation, and lease creation.

Level Rule Behavior
symbol If a function is already exact 100%, it is not a normal worker target. Exclude from candidate extraction and graph ranking edit targets.
file If every function/data item owned by a source file is exact, the file is read-only complete. Expose as reference evidence but never include in normal worker write_set.
lease A worker may only edit files with editability = editable. Lease creation rejects complete files, locked files, and files blocked by graph policy.
exception Maintenance or review cleanup on complete files requires explicit operator/admin mode. Not available to autonomous workers or automatic refill.
Current gap. The existing candidate code already skips symbols with fuzzy >= 100. The new design adds a file-level completion guard so a fully matched source file cannot be accidentally leased for edits just because it is connected to useful graph evidence.
Reference use. Complete files remain searchable matched references. The worker can pull whichever complete files it needs through graph and source searches, but those files stay out of normal write sets.

15. High-Level Implementation Checklist

This is the setup sequence for getting from the current loose knowledge layout to a working CLI-first graph/resource system. The first useful milestone is not every source being perfect; it is a working file card backed by code metadata and PR synthesis.

Phase Checklist Done When
0. Scaffold Add knowledge/sources, tools, and knowledge/resource_graph. Add registries, README files, and placeholder descriptors for the initial sources. The tree exists, source IDs are stable, and every slice has a descriptor.
1. Schemas Define source descriptors, search results, graph entities, graph facts, graph edges, rank features, and file-card JSON. Generated records can be validated before entering SQLite.
2. Graph Store Create SQLite schema, FTS tables, version tables, graph delta tables, and simple import/export commands. kg:rebuild-graph can create an empty valid graph database.
3. Code Graph Index current checkout metadata: source files, units, functions, symbols, splits, objdiff data, report status, and editability. kg:file-card --source <path> returns local match status and editability.
4. PR Source Wrap the existing PR dump, index changed files, PR text, diffs, reviews, and postmortems, then run the secondary synthesis pass. File cards include touching PRs, review risks, tactics, and fresh PR edges.
5. CLI APIs Add kg:sources, kg:status, kg:search, kg:file-card, kg:rank-features, and JSON output for each. Workers and operators can query the same information from shell commands.
6. Worker Boot Add file-card lookup to worker packets and prompts. Keep resource budgets bounded and expose linked files, PRs, risks, and resource edges. Every worker starts from a file graph card before deeper searches.
7. Ranking Merge graph rank features into board candidate scoring with deterministic v1 weights and complete-file editability guards. The director sees graph-aware candidates and cannot lease read-only complete files.
8. Worker Feedback Record worker facts, negative results, validations, retained edits, and graph updates after the no-regression gate passes. Worker output adds durable graph evidence and future ranking signals.
9. Extra Sources Add lightweight source slices for Discord, data sheet, PowerPC docs, external mirrors, and tool outputs as needed. Optional sources are searchable through CLI APIs without becoming hard dependencies.
10. External Tools Add tools wrappers for high-value tool calls such as Ghidra, opseq, mismatch-db, and MWCC debug. Tool outputs can be cached, searched, and optionally folded into graph facts.
11. Refresh Loop Wire PR refresh, source reindexing, graph delta records, and events that wake ranking when new edges appear. Merged PRs and refreshed sources update file cards and candidate scores.
12. Operations Add smoke tests, stale-source checks, graph rebuild checks, CLI help, and small fixture-based validation for each source/tool slice. The system can be rebuilt and smoke-tested without running a full decomp campaign.
Suggested first milestone. Build phases 0 through 5 with only code_graph and past_prs. That creates the first real file graph card and keeps the external sources out of the critical path until the core loop is useful.

16. Current V1 Implementation

The first milestone is implemented as a CLI-first TypeScript graph layer. It builds a SQLite graph from code_graph, past_prs, and graph-owned enrichments such as agent_shared_state and curator_enrichment, exposes file cards/search/rank features, and keeps other sources/tools registered as optional slices.

  • knowledge/sources/ contains source descriptors plus visible data, indexes, api, commands, and tests folders for code graph, past PRs, Discord, data sheet, PowerPC docs, external mirrors, and tool outputs.
  • Each source slice has data/ so the real corpus paths are visible without moving large or already-established resource trees.
  • tools/ contains external tool descriptors, lightweight CLI API stubs, runner/cache/index/test folders, and descriptors for Ghidra, opseq, mismatch-db, and MWCC debug.
  • knowledge/resource_graph/ contains graph docs, placeholders, ignored SQLite graph outputs, graph command notes, and internal enrichment artifacts.
  • packages/knowledge/src/graph/ implements graph schema, rebuild, code indexing, PR indexing/synthesis, legacy agent-state enrichment, internal curator ingestion, file cards, search, and rank features.
  • kg-* CLI commands expose source listing, graph status, legacy agent-state import, curator reduction, maintenance, graph rebuild, search, file-card, and rank-feature output.
  • Board snapshots optionally consume graph rank bonuses and filter read-only complete files when the graph DB exists.

17. First Implementation Slices

  1. Registry and schema. Add resource descriptors and graph tables. Keep existing prompt manifests separate from the resource registry.
  2. Resource-slice scaffolding. Add knowledge/sources/<source_id> directories with descriptors, data or data pointers, indexes, APIs, commands, and smoke tests.
  3. External-tool scaffolding. Add tools/<tool_id> directories for callable integrations such as Ghidra, opseq, mismatch-db, and MWCC debug.
  4. PR graph indexer. Ingest changed files, diff lines, review comments, text corpus, and postmortems into file and PR entities.
  5. PR synthesis pass. Group PR records by touched files, symbols, subsystems, review risks, accepted tactics, failed tactics, and fresh graph deltas.
  6. Code graph indexer. Build file, unit, symbol, function, match-status, and editability entities from repo metadata and reports.
  7. File card API. Expose kg:file-card and use it in worker packets as the first required lookup with linked files, recently updated evidence, PR history, resource edges, and editability.
  8. Document resource slices. Add Discord knowledge, PowerPC PDFs, and data sheet search as separate vertical slices.
  9. Ranking features. Feed file-card features into board candidate scoring and enforce read-only complete files before lease creation.
  10. Merge refresh loop. When PRs merge upstream, refresh PR data, rerun synthesis, record graph deltas, and wake ranking for files with new edges.

18. V1 Decisions

  • Expose resource access as lightweight CLI commands and package-local functions, not a full MCP server.
  • Treat external sources as optional and snapshot-based unless a concrete query or operator action asks for refresh.
  • Automatically accept most worker-provided facts when validation passes and no local regressions are retained.
  • Use deterministic ranking weights in v1 while logging enough feature/outcome data to support learned weights later.
  • Let workers request complete matched files as references on demand; do not preload a fixed set beyond the file-card budget.
