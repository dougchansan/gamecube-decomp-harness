Sidebar Flow Design

Proposal for restructuring the dashboard's left rail around the operating cycle of a decomp session, instead of a flat list of always-visible control panels. The core idea: the sidebar is a phase stepper. The system derives the current phase from durable state, surfaces one primary next action, and visibly locks the actions that must not run in the current phase (e.g. no PR intake while workers are running, no scheduling while handoff is in flight).

1 · The Operating Cycle

Each pass through this loop is one run session — one grouping of changes measured against one frozen baseline. Sync exists for two reasons: it catches duplicate work (upstream may have matched a function we also hold locally) and it keeps a known-good recovery point if our changes break something. The loop never deletes local evidence: carry-forward items (clean non-exact improvements, fact requests, stalls) survive checkpoint and fresh-run and re-enter the next session.

2 · System Map

The map is built around two ideas. First, the knowledge core is the hub: the resource graph and its feeders sit in the center, and everything else either feeds it (worker reports down, merged PRs up from sync) or reads it (ranked board and knowledge_context back up into the run loop). Second, the operational cycle reads clockwise by number: ① run across the top — workers are the main thing happening — then when the goal is reached, ② pause → validate → ship down the right column, then ③ sync along the bottom flowing right-to-left, closing the loop up the left edge into the next session. Colors: blue control/scheduling, green evidence, violet knowledge, amber git/baseline/ship. The hard lock means only one numbered stage is ever active at a time.

The quick read: the trigger actor (a plain process, not an agent) watches the board, wakes the director for one turn at a time, and starts workers under leases. Workers edit the checkout and write everything they learn back to the board — reports, facts, negative results. The score gate is the only thing that moves the baseline. In the background, the knowledge curator reduces worker reports and PR postmortems into lessons, a deterministic rebuild ingests them into the resource graph, and the graph flows back in as ranked priorities for the trigger and knowledge_context in worker packets — that's the full circle: workers produce evidence, curation turns it into knowledge, knowledge makes the next worker smarter. When the run goal is reached the cycle leaves the top band: handoff (②) drains, checkpoints, QA-validates with the reconcile agent fixing regressions, and ships; sync (③) pulls upstream back in, reconciles the ledger onto the new master, and the left-edge arrow closes the loop into the next session. This map supersedes the old mermaid diagram as the canonical visual.

3 · The Reconcile Agent (new)

Workers improve match coverage inside leases; the director schedules; the PR-review and curator agents handle knowledge. Nobody currently owns "make this bundle safe" — safe to ship upstream, and safe to continue from after upstream moves. That is the reconcile agent: an operator-triggered Pi agent beside the PR-review agent, outside the worker lease loop, sharing the same runner-owned validation gates. It runs only while scheduling is locked (run paused or no session active) and works the build until the gate is green or it escalates.

Ship-validate loop (Phase 4 → 5)

drain→ checkpoint→ build + regression report→ reconcile: fix regression→ rebuild · re-report ⟲ until clean (budgeted) → split plan→ merge latest master→ reconcile: conflicts · errors→ push PR

Sync-merge loop (Phase 6 → 2)

pull origin/master→ intake merged PRs→ replay carry-forward→ reconcile: conflicts · dupes · build errors ⟲ until clean build → new baseline→ new session

The lock around all of this is hard, not advisory. Once a sync starts, the old session is closed — there is no resuming the old session against the old baseline. While the reconcile agent holds the checkout, director/worker scheduling is refused server-side, the same way worker and trigger commands already reject non-active runs.

4 · Sessions and the Local Change Ledger

A run session is a grouping of changes measured against one frozen baseline. Shipping moves a subset of that grouping upstream; syncing closes the session and opens the next one against the updated baseline. The local change ledger — carry-forward patches, facts, lessons, rejected hypotheses — is the part of the ecosystem that persists across sessions.

SESSION N
baseline Bₙ = master @ a1b2c3
  • Runs accumulate changes c1 … c9
  • Checkpoint splits the grouping:
  • PR candidates {c1 c2 c4} → shipped
  • Carry-forward {c3 c5–c9} → ledger
SYNC BOUNDARY
session closes · hard lock
  • master moves: our PRs + others' merge
  • Reconcile: replay ledger onto new master
  • Dupes resolved: upstream wins, facts kept
  • New baseline Bₙ₊₁ captured
SESSION N+1
baseline Bₙ₊₁ = updated master
  • Starts already holding {c3 c5–c9}′
  • Board re-ranked with new graph facts
  • Runs continue from the updated state
The ledger is the continuity: only shipped PR candidates ever leave the system; everything else flows into the next session

5 · Phase Derivation (no new state required)

The phase is computed, not stored. Everything below already exists in the dashboard payload.

These locks are hard and server-enforced, not UI suggestions: (a) sync/intake is rejected the moment a run is active and stays rejected until the bundle ships or the operator explicitly discards it — pulling upstream mid-run would invalidate the baseline all worker evidence is measured against; (b) handoff and reconcile actions are rejected while anything holds a lease, so validation always sees a quiet checkout; (c) a started sync closes the session — there is no path back to the old baseline.

7 · Sidebar Proposal

The rail keeps its three structural pieces (Project disclosure at top, Process panel at bottom) but the middle becomes phase-driven: a persistent stepper, then one expanded panel for the current phase with its primary action, with other phases' panels collapsed and lock-annotated. Settings (agent, scheduling, QA/split inputs) stay inside their phase's panel — you meet each setting when it becomes relevant.

What stays the same

Project disclosure stays pinned at the top of the rail above the stepper; the Process panel (PID, saved processes, logs link) stays pinned at the bottom — both are ambient, not phase-bound. All existing endpoints are reused as-is; this is a presentation + gating layer, not a new API. A "show all controls" escape (the current flat layout) can remain behind an advanced toggle for debugging, since the CLI keeps single-step commands for the same reason.

8 · Open Questions

#QuestionLeaning
1How does the system know Phase 5 → 6 (PRs shipped)? It never publishes PRs itself. Explicit operator confirmation ("Mark shipped"), cross-checked against gh pr status when available.
2Reconcile attempt budget — how many fix → rebuild → re-report cycles before it stops? 3 attempts per regression, then a durable incident report for the operator; same repair-loop shape as the worker post-return gate.
3Duplicate policy on sync: upstream matched a function we also hold locally — who wins? Upstream wins by default; the local attempt is preserved as graph facts/lessons, not as code.
4Is reconcile a new top-level agent or a worker capability? New top-level agent beside PR-review: it needs whole-checkout authority (merge, rebuild, multi-file fixes) that the lease/write-set model deliberately denies workers.
5Where does "behind origin/master" freshness come from? Cheap fetch + ref compare on the dashboard interval, or last-intake timestamp in /api/config.
6Mid-run upstream merge of a file a worker is editing — surface it? No: the hard freeze means the run completes against its baseline; the conflict surfaces at the sync boundary, where the reconcile agent owns it.
