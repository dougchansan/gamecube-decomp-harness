Decomp orchestrator design

doldecomp/melee · proposed system before implementation

  • This document defines the top-level decomp-orchestrator system we want to build for Melee decomp.
  • It should consolidate the former decomp-find and melee-decomp surfaces into one top-level run system, with sweep tooling kept as a last-resort helper mode:
    • Those surfaces become internal scoring logic, agent context, prompts, and CLI helpers inside one larger system.
  • It should still rely on the repo's hard tools:
    • configure.py, ninja, objdiff, reports, locks, optional workspaces, and artifacts.
    • The thing directing the run is a central run-director Pi agent that understands the whole board, delegates bounded work, and wakes back up when the board changes.
Core principle.
  • Treat decomp like Sudoku.
  • Do not camp on one square just because it is unfinished.
  • When one file yields a fact, a source shape, a field name, or even a strong negative result:
    • Ask where else on the board that new constraint makes progress easier.
  • Concrete Sudoku example:
    • Finding a 9 somewhere on the board does not always mean you can immediately finish the square you were focused on.
    • It can still remove 9 as a possibility from a row, column, or box.
    • That removal may make a different square deterministic.
    • The decomp equivalent is a worker discovering a struct field, source shape, duplicate pattern, or negative result that does not finish its current file but makes another target much easier to fill in.
Agent runtime.
  • All reasoning agents in this system should be Pi agents launched through the Pi agent SDK.
  • The decomp orchestrator itself is not the main agent:
    • It is a thin stateful runner/server that stores facts, leases, events, and artifacts.
    • It creates or resumes director Pi-agent sessions and creates worker Pi-agent sessions only when there is work to do.

01 Goals

use the whole board, parallelize evidence, fill the next constrained piece

  • Build a system that can intelligently fill in and decompile pieces by using the full project data:
    • Progress reports.
    • Objdiff signals.
    • Source conventions.
    • Past PRs.
    • Duplicate groups.
    • Worker results.
    • Accepted facts and rejected hypotheses.
  • Parallelize the search for useful evidence:
    • Many workers can explore different files, symbols, patterns, and hypotheses at once.
    • Each worker may produce a match, a partial improvement, a fact, or a negative result.
    • Those outputs become shared constraints for the next scheduling cycle.
  • Prefer the easiest or most constrained useful pieces first:
    • Do not brute force through one singular file just because it is currently selected.
    • Find the targets that are easiest to improve with the information we already have.
    • Fill those in, then use the new information to make educated guesses about what else should become easier.
  • Use global matched_code_percent as the north-star metric:
    • The long-term target is full decompilation: exact matched code pushed toward 100%.
    • Intermediate targets like +1.0% matched_code_percent are run checkpoints and stop conditions.
    • fuzzy_match_percent and linked/complete progress are secondary telemetry, not the objective.
    • Do not chase fuzzy score as a success metric; it is already high compared with real matched/decompiled code.
  • Treat each run as the unit of progress:
    • A worker target, source file, or symbol is not a PR boundary.
    • A run can aim for a batch goal such as +5.0% matched_code_percent.
    • The orchestrator keeps integrating verified improvements into the run baseline until the target is reached or useful evidence runs out.

02 Architecture

director reads board state; workers do the research and source work

  • The director's world is intentionally small:
    • It reads the indexer output, reducer output, and run target.
    • It then writes decisions into shared run state.
  • The state substrate owns:
    • Leases, events, locks, target packets, artifacts, and wakeups.
  • Workers pull bounded leases from that state:
    • Do the PR search, resource lookup, source archaeology, optional last-resort sweep/permuter work, and cleanup inside their leased write sets.
    • Return reports and events for the reducer/director loop.

03 Run Director Loop

wake, read the board, delegate, sleep until the board changes

  • The run director is the central Pi agent:
    • It is not a permanent process thinking every second.
    • It sleeps between events.
  • The state substrate and runner/server wake it at scheduling turns:
    • Run start.
    • Worker completion.
    • Worker stall.
    • Score integration.
    • Regression.
    • When active workers drop below the desired count.
  • Each cycle:
    • Reads the current board.
    • Asks what new information changed the constraints.
    • Delegates the next best target packet to an available worker.
  • This layer should be agentic rather than a fully deterministic assignment engine:
    • Deterministic helpers calculate features, enforce locks, refresh reports, and verify patches.
    • The director uses those facts to make an informed delegation decision over the whole project.
  • Workers do not talk to each other directly:
    • A worker Pi agent writes its state, artifacts, facts, blockers, and final report to durable storage.
    • It then emits a wake event.
    • The next time the director Pi agent runs, it absorbs that state and decides what should happen next.
  • The runner/server can stay alive to supervise child processes and receive events:
    • It is deliberately boring: no broad reasoning loop, no project strategy, and no hidden always-on director.
    • Its job is to notice that something needs to happen, invoke the right Pi-agent session, persist the result, and go quiet again.

Director agent cycle

  • The director does not perform source research.
  • It only sees enough to decide where effort is most influential:
    • Run target.
    • Candidate files and their current matched-code state.
    • Indexer/reducer facts.
    • Active lease, stall, lock, and integration state.
  • Its output is a bounded scheduling decision, not a decomp attempt.
  1. Board read. Load current progress, active leases, worker reports, accepted facts, rejected hypotheses, duplicate groups, and recent stalls.
  2. Constraint propagation. Ask what the new facts make easier elsewhere: same assembly shape, same callback table, same struct field, same PR pattern, same mismatch class.
  3. Delegation. Fill idle worker slots with bounded target packets. A 16-worker run should refill when it falls to 15 active workers unless the run target is already reached.
  4. Sleep. Go idle after writing decisions. Worker Pi agents operate independently, write durable state, and wake the director only when they finish, stall, need a fact, or produce a score candidate.

Worker report contract

  • Workers are useful even when they do not land a perfect match.
  • A worker report should tell the director whether the board gained:
    • A patch.
    • A fact.
    • A negative result.
    • A cleaner understanding of why this target is not ready yet.

Wake events

  • The events table is the handshake between worker Pi agents and the sleeping director Pi agent.
  • A worker never messages another worker and never needs the director to be live while it is working.
  • It writes an event row such as:
    • worker_finished
    • worker_stalled
    • needs_fact
    • score_candidate
    • pool_below_target
  • The trigger actor writes pool_below_target when queued work is low, schedulable distinct-file work is low, queued work is blocked by active file locks, long-tail drain persists, or an operator periodic replan interval fires.
  • pool_below_target events are prioritized ahead of ordinary worker-report backlog so the director can refill capacity before the run drains into a slow tail.
  • The orchestration runner sees an unhandled event:
    • Invokes or resumes the director through the Pi agent SDK.
    • Marks the event handled after the director writes its next decisions.

Fact types

  • asm_duplicate_group: normalized assembly group with matched refs and unmatched candidates.
  • source_shape_pattern: declaration order, branch shape, helper inline shape, loop form, or pragma behavior that improved a target.
  • type_or_field_hint: struct field, accessor, callback signature, table type, or data ownership fact.
  • negative_result: a capability or hypothesis class that repeatedly preserved the bad first mismatch or caused regressions.
  • integration_fact: accepted patch, score delta, validation commands, and affected graph neighbors.

05 Board Prioritization

helper scores inform the director; the director chooses the next square

  • The board can produce deterministic features.
  • The final scheduling choice should be made by the run director.
  • The helper score is a candidate prior:
    • It surfaces places where a worker is likely to create useful information.
    • It does not just surface places where a local function might hit 100%.

06 Worker Lifecycle

lease, attempt, verify, emit evidence, release

Target packet

  • The director should send each worker a bounded packet:
    • Unit.
    • Symbol or file.
    • Source path.
    • Write set.
    • Baseline metrics.
    • Accepted facts.
    • Rejected hypotheses.
    • Enabled capabilities.
    • Verification command.
    • Budget.
    • Stop condition.
  • The worker then performs the PR, docs, source, naming, and resource research inside its lease.

Worker agent cycle

  • The worker is where the messy decompilation context belongs.
  • Worker read set should include:
    • Target packet, baseline metrics, write-set lock, accepted facts, and rejected hypotheses.
    • report.json, objdiff.json, target assembly, current source, translation-unit preamble, sibling functions, and nearby headers.
    • symbols.txt, r13/r2 small-data references, type definitions, struct member hints, and module-specific context.
    • Past PRs, local docs, decomp.me public scratch status, previous run bundles, attempt history, and reducer facts.
  • Worker outputs should always be reducer-friendly:
    • Patch candidate or no-patch result.
    • Objdiff delta and validation transcript.
    • Accepted facts, rejected hypotheses, missing context requests, and stall reason.
    • Score candidate event or worker-finished/stalled event.

Research/read phase

  • The worker should build a compact context pack before making source edits:
    • Read report.json and objdiff.json for the unit, source path, base object, match score, size, and diff command.
    • Extract target assembly and current source for the target function.
    • Extract the translation-unit preamble, sibling functions, file-scope variables, static data, macros, and local coding conventions.
    • Scan nearby headers and module headers for types, fields, callbacks, enums, and constants.
    • Resolve exact globals from symbols.txt, including r13/r2 small-data references.
    • Search past PRs, local docs, previous run bundles, reducer facts, and decomp.me public scratches for matching source shapes or known attempts.
  • The worker should turn missing context into explicit fact requests:
    • NEED_TYPE for a struct, typedef, enum, or callback signature it cannot justify.
    • NEED_SYMBOL for an exact global/data/function label from symbols.txt.
    • needs_fact report when the target is blocked until the board learns that fact.
  • The worker should preserve attempt history:
    • What was tried.
    • Which candidate compiled.
    • Which candidate improved or regressed objdiff score.
    • Which mismatch key persisted across source-hash-distinct candidates.
    • The best source shape so far.
  1. Receive target packet. The run director Pi agent chooses the target and the state substrate locks its write set, records base revision, creates an optional workspace if needed, and writes the packet.
  2. Start Pi session. The runner/server creates a worker Pi-agent session through the Pi agent SDK with the target packet as the user prompt and the leased workspace or checkout as cwd.
  3. Gather context. Before changing source, the worker searches relevant PRs, docs, source siblings, headers, symbols, splits, data sheets, naming conventions, public scratches, previous run bundles, and known mismatch patterns.
  4. Bootstrap workspace. Worker runs python configure.py if needed, then builds the narrowest relevant object/report baseline.
  5. Attempt planning. Based on that research, the worker chooses which capabilities are likely to produce progress or useful information.
  6. Attempt loop. Worker may combine focused editing, duplicate adaptation, fact research, cleanup validation, and optional last-resort sweep/permuter work. The loop pursues grounded hypotheses, not unlimited random variants.
  7. Verify narrowly. Use symbol or unit objdiff first. Only candidates that improve or reveal strong evidence get broader checks.
  8. Emit shard. Worker writes patch, result JSON, diff text, compile logs, and learned/rejected patterns into its artifacts.
  9. Report or stop. If progress continues, renew the lease. If evidence runs out, report "stalled/no useful guess," preserve context, release the slot, and emit a director wake event.

Stall policy

  • The target packet gives default attempt and wall-clock budgets.
  • The worker decides when to stop inside that budget based on the shared worker prompt.
  • Stop when the worker cannot name an evidence-backed next hypothesis.
  • No matched-code improvement after a fixed attempt budget or wall-clock budget.
  • Same first mismatch key repeats across many source-hash-distinct candidates.
  • Compile failures or reviewability failures exceed a threshold for the active capability.
  • The worker cannot name a plausible next hypothesis from PRs, docs, local source, duplicate groups, or measured diff signal.
  • A stronger related fact arrives; the worker may use it if helpful, but the director does not cancel active workers just because the board changed.

07 Write Safety

central file leases first; isolated workspaces only where useful

  • The mandatory safety primitive is the central SQLite lock, not a particular checkout layout:
    • Before editing, a worker receives a lease with an explicit write set.
    • The state substrate records file-level locks for every path the worker is allowed to touch.
    • A worker must not edit a file that is not covered by its active lease.
  • Worktrees or isolated workspaces are implementation options:
    • Use them when shared build output, generated files, risky headers, or long-running validation make one checkout too fragile.
    • A simpler first version can still be correct if the database owns edit locks and shared validation is serialized.
  • The main checkout remains the canonical integration surface.

09 Score Integration Gate

verified wins update the run baseline; PR packaging is outside the loop

  • The orchestrator does not create a PR per file, symbol, worker, or lease.
  • It accumulates verified improvements inside a run until the target matched_code_percent delta is reached.
  • Committing or packaging the final result for review is a separate human-facing step after the run produces a coherent improvement bundle.
  1. Confirm the worker still owns the lease and write set.
  2. Check the patch applies cleanly to the current canonical checkout.
  3. Run the narrowest relevant objdiff command and compare against the baseline attempt row.
  4. Run neighbor, unit, or full report checks if the patch touches headers, statics, data, splits, pragmas, or shared declarations.
  5. Record accepted facts and update graph edges affected by the integration.
  6. Regenerate build/GALE01/report.json, update run metrics, and ask the director to recompute the board unless the run target is reached.

End-of-run summary

  • When the run stops, produce a PR-description-style summary artifact.
  • It should include:
    • Baseline and final matched_code_percent.
    • Files changed.
    • Score integrations accepted.
    • Facts learned and rejected hypotheses.
    • Targets that stalled or should be revisited.
    • Validation commands and final report path.

11 Unified Skill Model

how the existing skills fold into the new top-level run system

  • The top-level orchestrator should feel like a single decomp run system rather than three separate skills with a wrapper around them.
  • The user starts one run:
    • Sets a target.
    • Chooses a worker count.
  • The run director Pi agent owns the board-level flow:
    • Reads indexer output, reducer output, and the run target.
    • Assigns the next most influential bounded target packet.
    • Wakes again when worker output or progress changes the board.
  • Duplicate adaptation, PR research, local cleanup, and optional last-resort sweep/permuter work are worker capabilities chosen inside a lease.
  • The old skills still matter, but they move down a layer:
    • decomp-find becomes board-scanning logic and ranking helpers.
    • melee-decomp becomes the worker system prompt plus co-located worker context.
    • melee-decomp-sweep becomes archived guidance plus last-resort sweep tools.
  • The top-level mental model is no longer "pick a target, then decomp it":
    • It is "update the board, learn a constraint, and ask where that constraint creates leverage next."

5Director-cycle dry run.Can invoke a director Pi agent on a board snapshot and produce a human-readable delegation plan without spawning workers.
6Prompt builder and capability templates.A shared worker system prompt is combined with target-specific initial user context from the packet, facts, budget, and enabled capability hints.
7One locked worker.Leases one target, acquires its file locks, optionally creates an isolated workspace, launches a worker Pi agent, emits an attempt artifact/report, and releases the lease.
8Score integration dry-run.Can verify a patch and explain accept/reject without applying it to the run baseline.
9Event-driven refill loop.When active workers drop below the target count, a wake event causes the runner to invoke the director Pi agent, which schedules the next worker.
10Fact-aware loop.Accepted and rejected facts change future director decisions and target packets.
11Human dashboard.Shows target matched_code_percent, current progress, active leases, Pi sessions, stalls, accepted facts, and score candidates.
12Run summary artifact.Produces a PR-description-style summary with matched-code delta, changed files, accepted integrations, facts learned, stalls, and validation evidence.

13 Implementation Defaults

settled assumptions for the first build

  • The system lives under top-level decomp-orchestrator/ in this repo.
  • It is not a Codex plugin and should not hide under tools/ as a side utility.
  • The primary objective is global matched_code_percent, with full decompilation as the long-term goal.
  • Runs are the progress boundary:
    • A run may target a batch gain such as +5.0% matched_code_percent.
    • Files, symbols, workers, and leases are work units, not PR units.
  • Central SQLite leases and file locks are mandatory; worktrees or isolated workspaces are optional implementation tools.
  • Header and data-owner locks start precise and become broader only when evidence shows dependent targets can be invalidated.
  • Active workers keep going after score integration:
    • The integration emits fresh facts/signals into shared state.
    • Workers may use those signals if they help, but v1 does not cancel or resume workers because the board changed.
  • Build/report generation is serialized in v1 through one global validation lock.
  • Crash recovery is restart-from-state:
    • Restart from the canonical codebase plus SQLite/artifact state.
    • Do not resume worker Pi sessions in v1.
    • Mark abandoned leases/workers stale and let the director schedule fresh work.
  • Worker prompting is standardized through a prompt builder:
    • Shared worker system prompt.
    • Target-specific initial user context generated from the target packet, facts, budget, and capability hints.
  • The initial score integration path is serial and evidence-producing; any auto-apply mode should be added later as an explicit policy.
  • The end-of-run artifact is a PR-description-style summary, not an automatic PR.
